# Architecture

This document describes how the AMPSphere frontend is put together: the stack,
the directory layout, the core data-flow patterns, and the conventions that any
change should follow. For build commands and a high-level overview see
`README.md`; for known gaps and planned work see `improvements.md`.

## Overview

The frontend is a single-page application (SPA) built with
[Elm Land](https://elm.land/) v0.20.1 (which sits on top of Elm 0.19.1 and Vite).
It is a **read-only client** over a public REST API:

```
https://ampsphere-api.big-data-biology.org/v1
```

There is no backend in this repository, no authentication, and no write
operations — every interaction is a `GET` that renders cataloged data
(antimicrobial peptides, families, search results, statistics).

The base API URL is configurable at runtime via the `API_BASE_URL` environment
variable; it is read in `src/interop.js` and passed to Elm as a flag.

## Tech stack

| Concern            | Choice                                                        |
|--------------------|---------------------------------------------------------------|
| Language / runtime | Elm 0.19.1 via Elm Land v0.20.1                               |
| Routing            | Elm Land file-based routing (`useHashRouting: false`)        |
| UI components      | `rundis/elm-bootstrap` 5.2.0 (Bootstrap 4.3.1 CSS via CDN)    |
| Charts             | Plotly.js (CDN) behind a `<plotly-chart>` web component      |
| Other visuals      | `<helical-wheel>`, `<copy-button>` web components (vanilla JS)|
| Custom styling     | `static/styles.css` (small; Bootstrap does the heavy lifting)|

latter holds Elm Land's generated `Main`, `Route`, `Page`, `View`, etc.

## Core patterns

### The `Effect` type and deferred API calls

Pages never return raw `Cmd`s; they return `Effect msg` (`src/Effect.elm`).
This is Elm Land's customized effect module, extended here with one important
case: **`ApiGet`**.

`Effect.apiGet` does not build the HTTP command immediately. Instead it stores a
function `makeCmd : String -> Cmd msg`. The base URL is injected later, in
`Effect.toCmd`, from `shared.apiBaseUrl`:

```elm
ApiGet apiOptions ->
    apiOptions.makeCmd options.shared.apiBaseUrl
```

This keeps the API base URL out of every call site and lets it be configured at
runtime. Each `src/Api/*.elm` module exposes a `get` that returns an `Effect`:

```elm
Api.Amp.get { accession = "AMP10.000_000", onResponse = GotAmp }
```

### Remote data

All asynchronous resources are modeled with `Api.Data`
(`src/Api.elm`):

```elm
type Data value
    = NotAsked
    | Loading
    | Success value
    | Failure Http.Error
```

Pages set fields to `Loading` in `init`, fire the `Effect`, and render the
result with `Api.view` instead of hand-writing the four-branch ladder:

```elm
Api.view
    { loading = Util.Html.spinner "Loading…"
    , failure = \_ -> Util.Html.errorAlert "Failed to load."
    }
    (\value -> …)
    model.data
```

`NotAsked` renders nothing; the `loading`/`failure` views are supplied per call
so each card can keep its own spinner style or message.

### API modules

Each endpoint gets its own module under `src/Api/` containing:

- `type alias` records for the decoded payload,
- a private `decoder`,
- a public `get : { … , onResponse : Result Http.Error a -> msg } -> Effect msg`.

Decoders are explicit and currently strict (a missing field fails the whole
response). Longer records use the `decodeAndMap`/`Decode.map2 (|>)` pipeline
idiom; smaller ones use `Decode.mapN`.

### Pages and routing

Routing is file-based: a module at `src/Pages/Foo/Bar_.elm` serves
`/foo/:bar`. Each page module exposes `page : Shared.Model -> Route params ->
Page Model Msg` and the standard `init` / `update` / `view` / `subscriptions`,
wrapped with `Page.withLayout (always <| Layouts.Default {})`.

### Layout, navbar, and global search

`Layouts/Default.elm` wraps every page with a Bootstrap navbar (with a global
search box) and a footer. The navbar's search submits to the **shared** update
via `Effect.sendSharedMsg`. `Shared.update` then routes the query:

- starts with `AMP` (no spaces) → `/amp/{id}`
- starts with `SPHERE` (no spaces) → `/family/{id}`
- otherwise → `/text-search?query=…`

The navbar owns `Navbar.State` and wires `Navbar.subscriptions` for the
collapse animation.

### Tabs with lazy loading

The AMP and Family pages use Bootstrap tabs. `Tab.State` is opaque, so the
active tab is tracked separately in the page model with explicit switch
messages; switching a tab can trigger the data fetch for that tab's content
(e.g. family features are only requested once needed).

### Pagination

Pagination is centralized in `Components.Pagination` (`view` and a compact
`small` variant). Pages pass `{ current, total, toMsg }` with **0-indexed**
pages (the user-facing label is `page + 1`); a page whose own state is 1-indexed
(TextSearch) converts at the call site. Use this instead of hand-rolling controls.

The footgun it encapsulates: pagination uses `<a href="#">`, and with
`Browser.application` a plain click on such a link is intercepted by the router
and pushes `#` onto history. The component attaches clicks with
`Util.Html.onClickPreventDefault` (a thin wrapper over
`Html.Events.preventDefaultOn "click"`), not `onClick`. Use that same helper for
any other action links.

### Web components for non-Elm visuals

Visualizations that are awkward in Elm are implemented as framework-agnostic
custom elements in `src/web-components/`, registered in `src/interop.js`
(`onReady`). Elm renders them with `Html.node` and passes inputs as `data-*`
attributes; the element owns its own DOM and re-renders on attribute change.

- **`<plotly-chart data-chart="…">`** — wraps Plotly.js. `data-chart` is a JSON
  blob (`{ data, layout, config }`) produced in Elm with `Json.Encode`.
- **`<helical-wheel data-sequence="…">`** — self-contained helical-wheel diagram
  (a vanilla-JS port of clemlab/helicalwheel; MIT, credited in-UI).
- **`<copy-button data-text="…">`** — copy-to-clipboard button with a fallback.

## Build & configuration

```bash
elm-land server   # dev server on http://localhost:1234 (debugger enabled)
elm-land build    # production build → dist/ (debugger disabled)
```

- `elm-land.json` configures HTML head (CDN links for Bootstrap CSS and
  Plotly JS), the `API_BASE_URL` env var, and the production/development
  debugger flags.
- There is no test framework configured yet; the build itself
  (`elm-land build`) is the compile-time safety net.

## Known inconsistencies (see `improvements.md`)

These are documented here so they aren't mistaken for intended patterns:

- **URL construction is mixed**: some `Api/*` modules use `Url.Builder`, others
  concatenate strings (which skips percent-encoding). `Api/Endpoint.elm` exists
  to centralize this but is underused.
- **Browse Data** does not reflect its filters/page in the URL (not
  bookmarkable), unlike TextSearch.
- **`/all_available_options`** returns ~29,500 microbial-source strings; with the
  dev-mode Elm debugger enabled, `Debug.toString` on this data overflows the
  stack. Production builds (debugger disabled) are unaffected.
